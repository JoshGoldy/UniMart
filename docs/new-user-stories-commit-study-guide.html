UniMart New User Stories Commit Study Guide

High-level revision notes for the actual 6-person commit order used for the newer UniMart features.

Purpose: this guide helps each teammate understand what happened in their commit, what files were affected, what the feature does, and how to explain it in a presentation, viva, or demo discussion.

Actual ownership order used: Hlaku committed 1st, Ndumiso 2nd, Josh 3rd, Elijah 4th, Ethan 5th, and Nkosi 6th.

Quick Feature Map

  • Commit 1: seller listing CRUD foundation in auth.js
  • Commit 2: live listing management create/edit UI in listings.html
  • Commit 3: delete and sold/active status controls in listings.html
  • Commit 4: listing image upload with Supabase Storage in auth.js and listings.html
  • Commit 5: clickable marketplace cards and listing details in search.html
  • Commit 6: forgot password and password recovery in auth.js and login.html

Commit 1: Hlaku

Seller Listing CRUD Foundation

Main file: auth.js Theme: Backend / Supabase logic

What happened: this commit added the core listing logic to the shared service layer so the app could create, update, delete, and fetch seller-owned listings from Supabase.

  • Added methods such as getMyListings(), createListing(), updateListing(), and deleteListing().
  • Created a consistent listing data shape so the frontend could render listings more easily.
  • Connected the app more deeply to the listings table in Supabase.

Why it matters: this commit is the foundation for all later listing features. Without it, the listing management page would still be static.

Simple explanation to give: “I worked on the backend-style service layer for listings. I added the JavaScript functions that talk to Supabase so sellers can create, fetch, update, and delete their own listings.”

Key things to remember:

  • auth.js acts like the app's service layer.
  • Supabase is called directly from the frontend through the JS SDK.
  • RLS policies still control which rows the user can actually access.

Commit 2: Ndumiso

Live Listing Management UI

Main file: listings.html Theme: Frontend form and page behavior

What happened: this commit replaced the old coming-soon listing management page with a real form and seller view where users can create and edit listings.

  • Added a create-listing form with fields like title, description, price, category, condition, and tradeable option.
  • Added rendering of the seller's listings on the same page.
  • Added edit mode so an existing listing can be loaded back into the form.

Why it matters: this is the first point where listing management becomes visibly usable to the user.

Simple explanation to give: “I turned the listings page from a placeholder into a working management page where sellers can add new listings and edit existing ones.”

Key things to remember:

  • The page uses form inputs and client-side validation.
  • The UI calls the methods in auth.js to persist changes.
  • The listing list is re-rendered after changes so the UI stays in sync with Supabase.

Commit 3: Josh

Delete And Sold Or Active Status Controls

Main file: listings.html Theme: Listing state management

What happened: this commit completed the seller-side management actions by allowing a user to delete a listing and mark it as sold or active.

  • Added action buttons on each seller-owned listing card.
  • Added logic for toggling listing status between active and sold.
  • Added delete confirmation and removal behavior.

Why it matters: this makes the listing management page practical, because sellers can now manage the full lifecycle of their items.

Simple explanation to give: “I added the management actions that let sellers delete listings and keep availability accurate by marking items as sold or active.”

Key things to remember:

  • Status matters because buyers should not see sold items as still available.
  • Delete and update actions still rely on the Supabase methods from commit 1.
  • This commit completed the seller control side of listing management.

Commit 4: Elijah

Image Upload Support For Listings

Main files: auth.js, listings.html Theme: Supabase Storage integration

What happened: this commit added real image upload support so sellers can attach images to their listings instead of just text details.

  • Added uploadListingImage() in auth.js.
  • Connected the upload flow to a Supabase Storage bucket called listing-images.
  • Stored the public image URL in the listings table.
  • Added preview behavior on the listing management page.

Why it matters: images make listings much more realistic and useful in a marketplace app.

Simple explanation to give: “I integrated Supabase Storage so sellers can upload real images, get a public image URL back, and save that URL in the listing record.”

Key things to remember:

  • The image itself is stored in Supabase Storage, not in the database table.
  • The database only stores the image URL.
  • This required both frontend form updates and backend-style storage logic.

Commit 5: Ethan

Clickable Marketplace Cards And Full Listing Details

Main file: search.html Theme: Buyer experience

What happened: this commit improved the marketplace page so users can click listing cards and open a fuller detail view before deciding whether they are interested.

  • Made cards interactive.
  • Added listing thumbnails to the marketplace cards.
  • Added the details modal showing more item information such as image, price, condition, description, status, and seller details.

Why it matters: this improves the buyer journey because users can inspect a listing properly before taking action.

Simple explanation to give: “I improved the marketplace browsing experience by making cards clickable and showing a more complete listing detail view.”

Key things to remember:

  • The search page is no longer just a simple card list.
  • The modal gives more context without leaving the page.
  • This commit is mainly about user interaction and presentation.

Commit 6: Nkosi

Forgot Password And Password Recovery

Main files: auth.js, login.html Theme: Auth recovery flow

What happened: this commit added a proper forgot-password workflow so users can request a reset email and then set a new password when they return from the recovery link.

  • Added requestPasswordReset() in auth.js.
  • Added completePasswordRecovery() in auth.js.
  • Expanded login.html to include a reset-email step and a new-password step.
  • Handled the Supabase recovery redirect and password update flow.

Why it matters: a real app needs account recovery so users are not locked out permanently if they forget a password.

Simple explanation to give: “I implemented the forgot-password flow by letting the user request a reset email and then complete the password reset through Supabase when the recovery link opens the login page again.”

Key things to remember:

  • Supabase Auth sends the reset email.
  • The app needs the correct redirect URL configured in Supabase.
  • The password update is done securely through Supabase Auth, not by manually storing passwords anywhere in our tables.
