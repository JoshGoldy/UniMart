UniMart UML Study Guide

High-level revision notes for the 4+1 architecture diagrams used to document the UniMart app.

What these diagrams are for: the UML set explains the UniMart system from different viewpoints, rather than trying to show everything in a single diagram. This follows the 4+1 model, where each view answers a different design question.

Simple way to remember it: the diagrams show who uses the system, what the main parts of the system are, how those parts interact, how the code is organized, and where the system is deployed.

The 4+1 Model In One Sentence Each

  • Scenarios view: shows what users want to do with the system.
  • Logical view: shows the main objects, services, and responsibilities in the design.
  • Process view: shows how actions happen step by step over time.
  • Development view: shows how the software is split into files, components, and dependencies.
  • Physical view: shows where the system runs and how the deployed parts communicate.

1. Scenarios Use Case Diagram

Scenarios View

This diagram shows the main goals of the system from the user’s point of view. It focuses on what a Guest User and a Registered User can do in UniMart.

  • Guest users can register, verify OTP, and log in.
  • Registered users can manage profiles, browse listings, create listings, edit listings, and view the dashboard.
  • The relationships such as <<include>> and <<extend>> show where one use case depends on or extends another one.

High-level meaning: this diagram explains the system in business terms. It is useful for showing which features matter to users before going into technical detail.

What to say in a test: “The use case diagram captures the main user goals of UniMart and shows how marketplace, account, and listing features are grouped around guest and registered users.”

2. Logical Class Diagram

Logical View

This diagram explains the internal structure of the app at a design level. It shows the major classes and their responsibilities.

  • Boundary classes represent the UI pages such as LoginPage, ProfilePage, and SearchPage.
  • Control class AuthService handles application logic such as authentication, profile updates, listing queries, and uploads.
  • Entity classes such as User, Listing, and DashboardAnalytics represent the core business data.

High-level meaning: this is the best diagram for showing how the app is logically designed. It separates user interface concerns, logic, and stored data.

What to say in a test: “The class diagram shows that UniMart follows separation of concerns, where pages handle interaction, the service layer handles logic, and entity classes represent the marketplace data.”

3. Process Sequence Diagram

Process View

This diagram shows how the system behaves over time for a few key workflows. In UniMart, it focuses on authenticating the user, creating a listing with an image, browsing the marketplace, and opening listing details.

  • The user interacts with the page.
  • The page calls AuthService.
  • AuthService communicates with Supabase Auth, Storage, and the database.
  • The final result is returned to the page and shown to the user.

High-level meaning: this diagram explains the runtime flow of the system. It is especially useful for describing how multiple services work together in one feature.

What to say in a test: “The sequence diagram shows the order of calls between the user interface, the app service layer, and Supabase services during important UniMart workflows.”

4. Development Component Diagram

Development View

This diagram shows how the software is organized into implementation pieces. It maps the frontend pages, shared assets, and external service dependencies.

  • The HTML pages represent the user-facing screens.
  • styles.css provides shared styling.
  • app.js provides shared UI behavior.
  • auth.js connects the frontend to Supabase for authentication, database queries, and storage.
  • The Supabase JavaScript SDK acts as the bridge to external backend services.

High-level meaning: this is the best diagram for explaining code organization. It shows how the project is modularized and which files depend on which services.

What to say in a test: “The component diagram shows the implementation structure of UniMart, especially how multiple pages reuse shared frontend assets and a common Supabase integration layer.”

5. Physical Deployment Diagram

Physical View

This diagram shows where UniMart runs after deployment and how the deployed parts communicate.

  • The user interacts with the app through a web browser on their device.
  • The static frontend is served from a hosting platform such as Azure or GitHub-based hosting.
  • The frontend communicates with Supabase Cloud for authentication, database access, and image storage.
  • The database side is protected by Row Level Security, which controls which rows a signed-in user may access.

High-level meaning: this diagram explains the runtime infrastructure of the app. It is useful for showing that UniMart is a static frontend backed by managed cloud services.

What to say in a test: “The deployment diagram shows that UniMart is hosted as a static web app while Supabase provides the cloud services for auth, database, and file storage.”

How The Diagrams Work Together

Big Picture
  • The use case diagram explains user goals.
  • The class diagram explains system structure.
  • The sequence diagram explains behavior over time.
  • The component diagram explains code organization.
  • The deployment diagram explains runtime infrastructure.

Together, they give a full architectural picture of the app. This is exactly why the 4+1 model is useful: each diagram answers a different question about the same system.

Quick Revision Points

Diagram Main question it answers Short answer for UniMart
Use Case What do users want to do? Register, log in, manage profiles, manage listings, browse listings, and view analytics.
Class What are the main logical parts? Pages, service layer, and core entities like users, listings, and analytics.
Sequence How do features execute over time? The UI calls the service layer, which then communicates with Supabase services and returns results.
