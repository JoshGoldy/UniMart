UniMart Product Backlog

Backlog aligned to the provided campus marketplace brief, with the current implementation status of the UniMart app as of 17 April 2026.

Purpose: This backlog turns the brief into a full product view of the app, grouped by requirement area. Each backlog item shows whether it is already in the app, partially implemented, or still to be completed.

Current build scope: UniMart currently includes student sign-up/login with OTP verification, profile editing, live listing management with image upload, marketplace browsing with search and category filtering, and a seller-facing dashboard with listing analytics. Major platform areas such as staff workflows, payments, messaging, ratings, and advanced analytics are still outstanding.

Backlog Status Summary

Completed Partially Completed Still To Be Completed

Completed Areas

  • Student registration, login, OTP verification, and session-based access control.
  • Student profile viewing and profile editing.
  • Listing CRUD with image upload, status, category, condition, and trade flag.
  • Marketplace browsing, keyword search, category filter, and listing detail modal.
  • Seller dashboard with live listing counts and charts.

Partial Areas

  • User verification uses Supabase Auth, but the required 3-role model is not complete.
  • Buyer experience exists for browsing, but condition and price-range filters are not complete.
  • Analytics exist for seller listings, but the brief requires broader reporting for moderation and trade facilities.

Still Outstanding

  • Trade Facility Staff and Admin workflows.
  • In-app messaging and negotiation chat.
  • Payments and partial-payment handling.
  • Ratings, reviews, trust, moderation, and seller reputation.
  • South African data integration and AI-assisted price suggestions.

Current Sprint Position

What the app can already demonstrate well

  • Secure account creation and sign-in through a 3rd-party auth provider.
  • Protected authenticated pages using session checks.
  • Profile maintenance for student users.
  • Live listing CRUD backed by Supabase database and storage.
  • Live marketplace browsing with real listing cards and listing details.
  • Basic seller dashboard analytics with charts.
  • Database protection through Row Level Security policies.

Recommended next backlog priorities

  1. Finish the buyer browsing experience by adding condition and price-range filters.
  2. Create the transaction domain so purchases, trade offers, and history become possible.
  3. Add messaging because it unlocks negotiation and supports later trade facility workflows.
  4. Implement the Trade Facility Staff and Admin role model with booking flows.
  5. Add ratings, moderation, and extended analytics once transactions exist.

This order keeps the current student marketplace flow moving forward while building the missing foundations needed for the more advanced brief requirements.
